Разберу каждую команду пошагово — с пояснением действий, зависимостей и нюансов:

**1. Запуск бэкенда:**
```bash
cd ../arepos-server && ./gradlew bootRun # backend на :8080 (нужен PostgreSQL)
```

Что происходит:
* `cd ../arepos-server` — переход в директорию проекта бэкенда (предполагается, что вы находитесь в соседней папке).
* `&&` — выполнение следующей команды только если предыдущая завершилась успешно.
* `./gradlew bootRun` — запуск Spring Boot‑приложения через Gradle Wrapper.

Важные нюансы:
* **Порт**: бэкенд стартует на `localhost:8080`.
* **Зависимость**: требуется запущенный PostgreSQL — приложение попытается подключиться к БД при старте. Если БД нет, приложение выдаст ошибку подключения.
* **Сборка**: `bootRun` автоматически собирает проект перед запуском (если изменений не было — использует кэш).
* **Остановка**: прервать выполнение можно сочетанием `Ctrl+C`.

---

**2. Запуск фронтенда:**
```bash
cd ../warchi && npm install && npm run dev # frontend на :5173
```

Что происходит:
* `cd ../warchi` — переход в папку проекта фронтенда.
* `npm install` — установка зависимостей из `package.json` (создаёт папку `node_modules`).
* `npm run dev` — запуск режима разработки (обычно через Vite/React/Vue).

Важные нюансы:
* **Порт**: фронтенд стартует на `localhost:5173`.
* **Кэширование**: `npm install` можно пропустить, если зависимости уже установлены и `package-lock.json` не менялся.
* **Автообновление**: режим `dev` отслеживает изменения в файлах и перезагружает страницу в браузере.
* **Логи**: в терминале будут выводиться ошибки сборки и предупреждения.

---

**3. Открытие приложения в браузере:**
```bash
open http://localhost:5173 # открыть wArchi
```

Что происходит:
* Команда `open` (для macOS) запускает браузер по умолчанию и открывает указанный URL.
* Для **Windows** используйте `start http://localhost:5173`, для **Linux** — `xdg-open http://localhost:5173`.

Важные нюансы:
* Убедитесь, что фронтенд уже запущен (команда `npm run dev` завершилась успешно).
* Если порт занят, фронтенд может предложить другой (например, `:5174`) — тогда измените URL.

---

**4. Опциональная проверка API:**
```bash
# Опционально: curl -s localhost:8080/api/v1/models | jq
```

Что происходит:
* `curl -s` — отправляет HTTP‑запрос к эндпоинту бэкенда (`/api/v1/models`) в тихом режиме (без индикаторов прогресса).
* `| jq` — передаёт ответ в утилиту `jq` для форматированного вывода JSON (красивое отображение структуры данных).

Важные нюансы:
* **Условие**: бэкенд должен быть запущен (`bootRun` выполняется).
* **Аутентификация**: если эндпоинт требует токен/куки, команда вернёт ошибку 401/403.
* **Установка jq**: если утилита не установлена, используйте `curl` без `jq` — ответ будет в «сыром» виде.
* **Альтернативы**: для проверки API удобнее использовать Postman или браузер с DevTools.

---

**Типичные проблемы и решения:**
1. **Бэкенд не запускается**: проверьте, запущен ли PostgreSQL. Убедитесь, что в `application.properties`/`application.yml` указаны верные настройки подключения.
2. **Фронтенд не видит бэкенд**: убедитесь, что фронтенд настроен на проксирование запросов к `localhost:8080` (например, через `vite.config.js`).
3. **Ошибки зависимостей**: удалите `node_modules` и запустите `npm install` заново. Для Gradle очистите кэш (`./gradlew clean`).
4. **Порт занят**: найдите процесс на порту (`lsof -i :8080`) и завершите его (`kill <PID>`).

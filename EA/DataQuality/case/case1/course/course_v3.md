# Слайд 2. Что такое качество данных — примеры из банковской практики

```mermaid
flowchart LR
    subgraph BAD ["НЕКАЧЕСТВЕННЫЕ ДАННЫЕ"]
        B1["Телефон: +7 ыыы-23"]
        B2["Клиент: Иван Иванов, Клиент: И. Иванов, Клиент: Ivanov I."]
        B3["Дата паспорта: 32.13.2020"]
        B4["Остаток по кредиту: -5000"]
    end
    
    subgraph GOOD ["КАЧЕСТВЕННЫЕ ДАННЫЕ"]
        G1["Телефон: +7 916 123-45-67"]
        G2["Один клиент = одна запись"]
        G3["Дата паспорта: 13.02.2020"]
        G4["Остаток по кредиту: 5000"]

    BAD -->|"Мероприятия: валидация, дедубликация, контроль форматов"| GOOD

**Банковский аналитик (операционист CRM):** Вы вводите телефон клиента, но система позволяет сохранить «+7 ыыы-23». Кредитный конвейер не может дозвониться — заявка отклоняется.

**Эксперт (Data Steward):** DAMA DMBOK2, глава 13, раздел 1.3 — шесть измерений качества. DQAF, Dimension 3 «Accuracy and Reliability».

# Слайд 3. Три линии обороны — общая схема

```mermaid
flowchart TD
    subgraph LINE1 ["1-я линия: Готовность DWH"]
        L1_READY["Операционный день закрыт?<br>Данные загружены?"]
        L1_OUT["Светофор готовности"]
    end
    
    subgraph LINE2 ["2-я линия: Качество источников"]
        L2_CHECKS["Проверки: полнота, одинаковость, формат"]
        L2_OUT["Светофор качества"]
    end
    
    subgraph LINE3 ["3-я линия: Финальный контроль"]
        L3_REPORT["Отчёт сформирован"]
        L3_OUT["Светофор валидации"]
    end
    
    LINE1 --> LINE2 --> LINE3
    L1_OUT -.->|"Красный - стоп"| LINE2
    L2_OUT -.->|"Красный - инцидент"| LINE3
```

**Банковский аналитик (кредитный инспектор):** Это как согласование кредитной заявки. Сначала проверяют документы, потом — правильность заполнения, потом — условия кредитного комитета.

**Эксперт:** DAMA DMBOK2, глава 13, раздел 5.2 — Three Lines of Defense. DQAF, Dimension 4 «Serviceability».

**Эксперт:** DAMA DMBOK2, глава 13 — Completeness. DQAF, Dimension 3.1 «Source data».

# Слайд 8. Вторая линия: Дата-стюард и инцидент

```mermaid
flowchart TD
    ANALYST["Аналитик видит красный"]
    ANALYST -->|Нажимает Сообщить| TICKET["Автоматический тикет в системе инцидентов"]
    
    TICKET --> STEWARD["Дата-стюард принимает инцидент"]
    
    STEWARD --> DIAG{"Диагностика: в чём причина?"}
    
    DIAG -->|Ошибка в данных| FIX_DIRECT["Исправляет данные"]
    DIAG -->|Проблема в процессе| ESCALATE["Эскалация Владельцу данных"]
    
    FIX_DIRECT --> TEST1{"Данные исправлены?"}
    TEST1 -->|Да| STATUS_GREEN["Статус зелёный"]
    TEST1 -->|Нет| FIX_DIRECT
    
    ESCALATE --> OWNER["Владелец данных принимает решение"]
    OWNER -->|Утверждает исправление| FIX_OWNER["Техническая команда чинит"]
    OWNER -->|Меняет порог| CHANGE_RULE["Корректирует правило"]
    
    FIX_OWNER --> TEST2{"Исправлено?"}
    TEST2 -->|Да| STATUS_GREEN
    TEST2 -->|Нет| FIX_OWNER
    
    CHANGE_RULE --> STATUS_GREEN
    STATUS_GREEN --> ANALYST_NEXT["Аналитик продолжает работу"]
```

**Банковский аналитик (кредитный инспектор):** Нажали «Сообщить» — пришёл Дата-стюард. Если проблема в одной записи — починил сам. Если системная — позвал начальника.

**Эксперт:** DAMA DMBOK2, глава 13, раздел 2.7 — Data Quality Operations. DQAF, Dimension 0.4 «Other quality management».

# Слайд 13. Итоговая схема (полная версия)

```mermaid
flowchart TD
    START["Кредитный аналитик<br>получает заявку в CRM"]
    
    subgraph LINE1 ["1-я линия: Готовность DWH"]
        L1["Проверка: закрыт ли операционный день?"]
        L1_R{"Красный?"}
        L1_WAIT["Ожидание загрузки данных из АБС"]
    end
    
    subgraph LINE2 ["2-я линия: Качество CRM/MDM"]
        L2["Проверка: полнота, согласованность"]
        L2_R{"Красный?"}
        L2_INCIDENT["Создание инцидента"]
        L2_STEWARD["Дата-стюард / Владелец"]
        L2_FIX["Исправление или изменение правил"]
    end
    
    subgraph LINE3 ["3-я линия: Валидация заявки"]
        L3["Финальная проверка: скоринг, лимиты"]
        L3_R{"Красный?"}
        L3_REBUILD["Отказ или доработка"]
    end
    
    END1["✅ Кредит одобрен"]
    END2["❌ Кредит отклонён"]
    
    START --> L1 --> L1_R
    L1_R -->|Нет| L2
    L1_R -->|Да| L1_WAIT --> L1
    
    L2 --> L2_R
    L2_R -->|Нет| L3
    L2_R -->|Да| L2_INCIDENT --> L2_STEWARD --> L2_FIX --> L2
    
    L3 --> L3_R
    L3_R -->|Нет| END1
    L3_R -->|Да| L3_REBUILD --> END2
```

**Банковский аналитик (кредитный инспектор):** Весь процесс от получения заявки до одобрения. Если данные не готовы — ждём. Если некачественные — зовём стюарда. Если финальная проверка не пройдена — отказываем.

**Эксперт (руководитель кредитного управления):** End-to-end DQ процесс с decision gates.

- DAMA DMBOK2, глава 13, разделы 1-5

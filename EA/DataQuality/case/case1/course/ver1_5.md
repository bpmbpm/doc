# Слайд 2. Что такое качество данных — примеры (ИСПРАВЛЕН)

```mermaid
flowchart LR
    subgraph BAD ["НЕКАЧЕСТВЕННЫЕ ДАННЫЕ"]
        B1["Телефон: +7 ыыы-23"]
        B2["Клиент: Иван Иванов, Клиент: И. Иванов, Клиент: Ivanov I."]
        B3["Дата паспорта: 32.13.2020"]
        B4["Остаток по кредиту: -5000"]
    end
    
    subgraph GOOD ["КАЧЕСТВЕННЫЕ ДАННЫЕ"]
        G1["Телефон: +7 916 123-45-67"]
        G2["Один клиент = одна запись"]
        G3["Дата паспорта: 13.02.2020"]
        G4["Остаток по кредиту: 5000"]
    end
    
    BAD -->|"Мероприятия: валидация, дедубликация, контроль форматов"| GOOD
```

**Пояснение к рисунку:** Слева — типичные дефекты данных в банке. Справа — исправленные данные. Между ними — мероприятия по повышению качества.

**Банковский аналитик:** Некачественные данные — это когда вы звоните клиенту по телефону, а там «ыыы».

**Эксперт:** Ключевые мероприятия: валидация на вводе (маски, regexp), дедубликация записей, контроль ссылочной целостности.

# Слайд 4. Первая линия: готовность хранилища — Рисунок 4А. Вертикальный светофор (ИСПРАВЛЕН)

```mermaid
flowchart TD
    RED_LIGHT["🔴"]
    YELLOW_LIGHT["🟡"]
    GREEN_LIGHT["🟢"]
    
    RED_LIGHT --> YELLOW_LIGHT --> GREEN_LIGHT
    
    RED_TEXT["День открыт. Операции идут."]
    YELLOW_TEXT["День закрыт. Идёт загрузка."]
    GREEN_TEXT["Данные загружены. Можно работать."]
    
    RED_LIGHT -.- RED_TEXT
    YELLOW_LIGHT -.- YELLOW_TEXT
    GREEN_LIGHT -.- GREEN_TEXT
```

**Пояснение:** Вертикальное расположение повторяет дорожный светофор. Красный сверху — стоп. Жёлтый — предупреждение. Зелёный снизу — разрешение.

**Банковский аналитик:** Красный — ещё нельзя строить отчёт. Жёлтый — потерпите 15-20 минут. Зелёный — всё готово.

# Слайд 6. Вторая линия: проверка полноты анкеты — Рисунок 6А. Механизм расчёта (ИСПРАВЛЕН)

flowchart LR

# Слайд 8А. Полная схема обработки инцидента (ИСПРАВЛЕН)

```mermaid
flowchart TD
    ANALYST["Аналитик видит красный"]
    ANALYST -->|Нажимает Сообщить| TICKET["Автоматический тикет в системе инцидентов"]
    
    TICKET --> STEWARD["Дата-стюард принимает инцидент"]
    
    STEWARD --> DIAG{"Диагностика: в чём причина?"}
    
    DIAG -->|Ошибка в данных| FIX_DIRECT["Исправляет данные"]
    DIAG -->|Проблема в процессе| ESCALATE["Эскалация Владельцу данных"]
    
    FIX_DIRECT --> TEST1{"Данные исправлены?"}
    TEST1 -->|Да| STATUS_GREEN["Статус зелёный"]
    TEST1 -->|Нет| FIX_DIRECT
    
    ESCALATE --> OWNER["Владелец данных принимает решение"]
    OWNER -->|Утверждает исправление| FIX_OWNER["Техническая команда чинит"]
    OWNER -->|Меняет порог| CHANGE_RULE["Корректирует правило"]
    
    FIX_OWNER --> TEST2{"Исправлено?"}
    TEST2 -->|Да| STATUS_GREEN
    TEST2 -->|Нет| FIX_OWNER
    
    CHANGE_RULE --> STATUS_GREEN
    
    STATUS_GREEN --> ANALYST_NEXT["Аналитик продолжает работу"]
```

**Пояснение:** Детальная схема от красного светофора до зелёного. Два пути: прямое исправление (Дата-стюардом) и эскалация (Владельцу данных).

# Слайд 10. Компромисс: почему зелёный не всегда 100% (ИСПРАВЛЕН)

```mermaid
flowchart TD
    subgraph BUSINESS ["Позиция аналитика"]
        B1["Хочу 100% качество"]
        B2["Все поля заполнены"]
        B3["Все проверки включены"]
    end
    
    subgraph REALITY ["Реальность"]
        R1["100% стоит денег"]
        R2["100% замедляет работу"]
        R3["100% невозможно от внешних систем"]
    end
    
    subgraph RESULT ["Компромисс"]
        C1["Зелёный = достаточно для задачи"]
        C2["Жёлтый = допустимо с контролем"]
        C3["Красный = недопустимо"]
    end
    
    BUSINESS --> REALITY
    REALITY --> RESULT
```

**Пояснение:** Установка порогов — компромисс между желаемым качеством и затратами.

**Эксперт:** Data Quality измеряется в степени пригодности для использования (Fitness for Purpose).
